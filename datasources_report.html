1. The Big Picture

The agent has three layers of data sources. We always try the cheapest one first:

  1. Layer 1 — Vendor registry A static list of 134 vendors in vendors.json. Used for "do you know vendor X?" and for aliases (e.g. "Aruba" ↔ "HPE Aruba Networking").
  2. Layer 2 — Local SQLite DB data/switches.db with three tables: switches (specs), firmware_versions (release notes), security_advisories (CVEs).
  3. Layer 3 — Live web fallback If a model is not in the DB, live_extract.py searches the web (Startpage → Mojeek → DuckDuckGo) and parses the official datasheet on the fly.

Everything runs at $0: no paid APIs, no LLM. Just HTML/PDF parsing.

4. Data Sources for Security Advisories (CVEs)

For login-gated vendors (Cisco, Juniper, Aruba, Arista, Dell, Fortinet) we cannot reach the changelog. We side-step the wall by pulling the same data from NIST NVD — the U.S. government's free, no-auth CVE database. Then we overlay CISA KEV to flag which CVEs are being actively exploited in the wild.

Primary CVE source

NIST NVD

services.nvd.nist.gov/rest/json/cves/2.0
  • No auth required.
  • Rate-limited: 5 req / 30s → fetcher sleeps 6.5s.
  • Query by CPE (vendor + product code).
  • Run: python3 fetch_firmware.py nvd
"Actively exploited" overlay

CISA KEV catalog

cisa.gov/.../known_exploited_vulnerabilities.json
  • Free public JSON feed.
  • Flips actively_exploited=1 on matching CVEs.
  • Run: python3 fetch_firmware.py kev

5. How It Is Doing

  • Speed: answering DB-cached questions in ~1 ms (target was <10 s).
  • Coverage: 134 vendors recognised; 18 scrapers wired up; ~237 switch models scraped.
  • Firmware: 130 firmware versions across 3 active fetchers (MikroTik, NVIDIA, Ubiquiti).
  • Security: 2,728 CVEs loaded, 68 flagged as actively exploited.
  • UI: 5-tab Streamlit app (app.py) + CLI (cli.py) — all working.
  • Tests: 81 pytest + 34 smoke + 19 fast-path tests all passing.

6. Issues We Faced & What We Did

Issue 1 — Big enterprise vendors block scrapers (Aruba, Dell, Juniper). Their sites are behind Akamai or other bot-protection. Even with browser User-Agent headers we get 403 / 406. No amount of code-fixing will work — they actively don't want robots.
What we did: Stopped trying to scrape their HTML. For specs they fall back to the live web search. For firmware/CVEs we go through NIST NVD instead, which gives us structured affected/fixed version ranges and CVSS scores. The old "login portal" dead-end message was removed.
Issue 2 — TP-Link product pages are JavaScript-rendered. A plain HTTP fetch got an empty shell with no specs.
What we did: Switched the TP-Link scraper to read tp-link.com/us/sitemap.xml for the model list, then fetch each static product page directly. No headless browser needed.
Issue 3 — DuckDuckGo search returns 202 (rate-limited / blocked). Originally the live fallback relied on DDG.
What we did: Made live_search.py try Startpage → Mojeek → DDG in order. DDG is now only a last resort and not relied on.
Issue 4 — No LLM means messy vendor PDFs can't be parsed into full specs. Some datasheets have specs in image tables or weird layouts.
What we did: Degrade gracefully — return the official datasheet URL, product image, and any features we could extract, rather than failing. Honest "not in KB" answers instead of fake numbers.
Issue 5 — NIST NVD rate limit is strict (5 requests per 30 seconds unauthenticated). Naive paging gets 429.
What we did: The NVD fetcher sleeps 6.5 s between pages and supports --nvd-vendor cisco for incremental top-ups instead of full re-pulls.
Issue 6 — Streamlit reruns on different threads, but SQLite hates cross-thread connections. Caused random "database is locked" / thread errors in the UI.
What we did: Open the SQLite connection with check_same_thread=False and cache the agent via @st.cache_resource.
Issue 7 — Many CVEs are theoretical and noisy. 2,700+ rows is too much for an operator to triage.
What we did: Layered the CISA KEV "known exploited" flag on top, so the UI can show the ~68 CVEs that are actually being weaponized in the wild first.
Issue 8 — DB used to get wiped when re-built from seed. A build_db.py rebuild deleted the ~237 scraped models.
What we did: Switched to in-place migration (ALTER TABLE for new columns, CREATE TABLE IF NOT EXISTS for new tables). Rebuilds from seed are now opt-in only.
