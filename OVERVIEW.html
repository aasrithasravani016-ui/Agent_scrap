1. What it is (in one paragraph)

Switch Spec Agent answers questions about network switches — "what are the specs of a Cisco Catalyst 9300-48P?", "compare two switches", "which switches support 400G?", "what changed in newer firmware?" — for any vendor, any model, in under 10 seconds, using only free tools (no AI models, no paid APIs, no monthly cost). Known models answer instantly from a local database; unknown ones are fetched live from the public web, then cached so they're instant next time.

2. How it works (simple)

1
You ask for a switch (by model, SKU, or a natural sentence).
2
Local database first. A small SQLite file holds ~240 switches. If it's there → answered in about 1 millisecond.
3
If not found → live web fallback. It searches the free web (Startpage → Mojeek → DuckDuckGo), opens the top vendor pages/PDFs, reads the specs out of them, and merges the sources with a confidence score.
4
It caches the answer into the database, so the next time the same model is asked it's instant.
5
Extras: it also grabs a product photo when available, and a Firmware Advisor tells you what changed (security fixes, new features, bugs) between your firmware version and the latest.

No AI is used anywhere. All "intelligence" is plain code: regex parsing, HTML/PDF table extraction, vendor-name aliases, and confidence-based merging.

4. What it does well (pros)

  • $0 forever. No AI, no API keys, no subscriptions, no cloud.
  • Fast. Cached/known answers in ~1 ms; live lookups ~4 s, then cached.
  • Any vendor, any model. Not limited to a fixed list — the live fallback handles the long tail and caches it.
  • Honest by design. When it doesn't know, it says so or gives the official link — it never invents specs or firmware notes.
  • Works offline for anything already in the database.
  • Tested & version-controlled. 81 unit tests + smoke test, all green; full git history on GitHub.
  • Extensible. Adding a vendor scraper or firmware source is a small, well-defined task.
  • Two interfaces. A web UI for people, a CLI for scripts/automation.

5. Limitations — the reason, and how it's handled

Every limitation below is a consequence of the "no AI, no paid services" rule. None of them is a hidden bug — each has a deliberate, honest handling.

Limit 1 No single dataset of "every switch" exists.

Why: switch specs live scattered across hundreds of vendor websites and PDFs. Nobody publishes one master file.

Solution: a hybrid — a curated seed + self-growing scrapers + an on-demand live web fallback that caches what it finds. Coverage grows with use.

Limit 2 Specs from messy PDFs are best-effort.

Why: without an AI model, code can only reliably parse clean spec tables. Marketing-style multi-product PDFs don't have a regular structure.

Solution: instead of guessing wrong numbers, it returns the official datasheet link + the features it could detect. (A free-tier AI extractor could be added later if that rule is ever relaxed.)

Limit 3 Dell, Juniper, HPE Aruba block the scraper.

Why: their sites use bot-protection (Akamai), redirects to login stores, or 403-block a plain client. A $0 client can't get in.

Solution: these vendors are served on-demand by the live web fallback (which uses search engines that do work), not by bulk scraping.

Limit 4 Some built-in scrapers are "registered but thin".

Why: vendor catalog pages change or are JavaScript-rendered, so hardcoded URLs go stale. (TP-Link was fixed by switching to its sitemap; others still need the same treatment.)

Solution: the framework knows the vendor; meanwhile the live fallback covers it. Each scraper can be repaired one at a time (the TP-Link sitemap method is the template).

Limit 5 Firmware advisor depth varies by vendor.

Why: only MikroTik publishes a clean plain-text changelog. Ubiquiti exposes versions+dates via a JSON API but no change text. NVIDIA's docs are JavaScript. Cisco/Juniper/Arista/Aruba require a login.

Solution, honest per vendor: MikroTik = full security/feature/bug diff; Ubiquiti/NVIDIA = "you are N versions behind, here's the official notes link"; login-gated = an honest portal link, never a fake answer.

Limit 6 Free search engines can rate-limit.

Why: they actively discourage scraping; under heavy use they may return nothing for a while.

Solution: three engines are tried in order, results are cached, and common models can be pre-loaded with the scrapers so live search is the exception, not the rule.

Limit 7 The database file isn't stored in GitHub.

Why: a built database is a regenerable artifact; standard practice is to keep it out of source control.

Solution: the repo carries the seed + scrapers + live fallback that rebuild it. (A data snapshot can be committed on request if a clone needs to be instantly populated.)

Limit 8 It does not "reason" or recommend.

Why: by design — it is a fetcher of facts, not an AI advisor. This keeps it $0 and trustworthy.

Solution: it gives you accurate specs and structured firmware diffs; the human makes the decision. This is a feature, not a gap.

7. How to use it

Use python3 on this machine. Run from the project folder.

cd /Users/aasritha/agent/switch-spec-agent

# Web UI (recommended) — opens at http://localhost:8501
python3 -m streamlit run app.py

# Command line
python3 cli.py "Cisco Catalyst 9300-48P"
python3 cli.py "Ubiquiti USW-Pro-48"
python3 cli.py "compare C9300-48P vs EX4400-48P"
python3 cli.py "which switches support 400G"
python3 cli.py --firmware "MikroTik CRS326-24G-2S+RM" 7.8

# Grow the database from vendor sites
python3 run_scrapers.py mikrotik
python3 fetch_firmware.py mikrotik ubiquiti cumulus

# Verify everything is healthy
python3 -m pytest tests/ -q          # 81 passed
python3 tests/smoke_test.py          # 19 passed
