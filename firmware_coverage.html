1. How the advisor decides what to return

You enter a switch model and (optionally) the firmware version you're running. The advisor walks through five sources of information in order. The first source that has real data wins. The remaining sources are never queried — they don't need to be.

StepSourceWhat it returns
1. Identify vendor134-row vendor registry — name aliases, sub-brand mappings, domain indexCanonical vendor name (e.g. "TRENDnet" not "trendnet"; "HPE Aruba Networking" not "Aruba"). Without this step the live web fallback would attribute results to the wrong vendor.
2. Try Tier 1Hand-built per-vendor changelog fetchers in firmware_fetchers.pyFull release-note diff — security fixes, new features, bug fixes, known issues, deprecations — bullet by bullet, per version.
3. Try Tier 2NIST NVD CVE feed (security_advisories table populated by fetch_firmware.py nvd)Severity-broken-down list of known CVEs for that vendor + NOS, with the earliest fixed version computed.
4. Try Tier 3Live web fallback (live_firmware.py) — free multi-engine search + on-page regexThe vendor's latest publicly-listed firmware version, the release date if mentioned, and the source URL as the release-notes link. Cached on success — next query is instant.
5. Try Tier 422 known login-gated vendor portals (Cisco, Juniper, Arista, Aruba, Dell, …)Honest "release notes behind vendor login" message + the portal URL.
6. Honest Tier 5None — terminal fallback"No public firmware source available for this vendor at $0 / no-LLM. Check the vendor's support site."

4. Issues we hit and what we did

The original advisor only worked for 3 vendors.

MikroTik, Ubiquiti, NVIDIA Cumulus had hand-built fetchers. Everyone else got "no data — run fetch_firmware.py <vendor>" but no such fetcher existed for them, so the suggestion was empty advice.

Added Tier 2 (NVD CVE feed) and Tier 3 (live web fallback) so every vendor in the registry gets a real attempt at real data. Cleaned the misleading "run the fetcher" message so it only appears when a fetcher actually exists.
NVIDIA Cumulus only enumerated one version.

The fetcher pulled the docs index page and regexed slugs — but the page only listed the latest. Every other Cumulus release was invisible.

Rewrote the fetcher to probe every cumulus-linux-NNN/Whats-New/ Hugo page (4.0–5.16). One pointer-row jumped to 51 real version rows with parsed New Features / Resolved Issues / Open Issues content.
MikroTik fetcher returned zero rows.

The fetcher hit a JS-rendered changelog index and parsed garbage. Also tripped on a channel-resolution bug where the "testing" channel had the same version as "stable" and overwrote its label.

Switched to MikroTik's plain-text mirror at upgrade.mikrotik.com/routeros/<ver>/CHANGELOG. Fixed channel-resolution to capture the stable head directly. 15 versions populated with full security/feature/bug bullets.
Ubiquiti was scraping a CSRF-protected SPA.

community.ui.com is a single-page app and rejected the scraper. 61 UniFi switch versions were unreachable.

Switched to Ubiquiti's public firmware JSON API at fw-update.ui.com/api/firmware?filter=eq~~channel~~release. 61 versions + dates pulled, no login. The API has no release-note text so we surface version + date + official notes link, never inventing bullets.
"239 CVEs known" was useless without severity.

The advisor reported total CVE counts but not how scary they were.

Now reports the CRITICAL / HIGH / MEDIUM / LOW breakdown for each vendor + NOS. "239 CVEs — 37 CRITICAL · 119 HIGH · 81 MEDIUM · 2 LOW" tells a very different operational story than a raw count.
The advisor told users to run fetchers that didn't exist.

The "public firmware vendor" list included Netgear and TP-Link, but no fetcher existed for them. The "no data" message suggested a command that did nothing.

Trimmed the public-firmware list to vendors with real fetchers (3). The "run fetcher" hint now appears only when the fetcher key actually exists in the registry. Long-tail vendors get Tier 3 (live discovery) or Tier 5 (honest no-source) instead.
NVD coverage was only 12 vendors.

The CPE pattern list in the NVD fetcher only covered the obvious enterprise names.

Expanded the CPE list to 37 entries covering MikroTik, Ubiquiti, D-Link, Zyxel, TRENDnet, Allied Telesis, MOXA, Westermo, Hirschmann, Phoenix Contact, Schneider, Lenovo, Sophos, Tenda, Linksys, NEC, Yamaha, ZTE, Ruijie + extra OS lines for Cisco/Juniper/HPE/H3C. After running the fetcher: 18 vendors with 3,028 CVEs (up from 12 / 2,728).
Long-tail vendors dead-ended at "no data".

Any vendor outside the 3 hand-built fetchers and the 12-then-18 NVD-covered ones produced a flat "no public firmware source" — even when the vendor had a perfectly reachable public firmware page.

New Tier 3: live_firmware.py mirrors the spec live fallback. Searches the web, regexes a version pointer and date, caches the result. Pre-populated for every vendor in the registry (134 vendors processed; 50 returned a real version pointer).
Some live-discovered "versions" were obviously not versions.

Without a model query, the regex sometimes picked up phone numbers, building codes, or file sizes that happened to match the pattern. The DB ended up with entries like "L-com 83.984375" (which is 215/256, not a firmware version).

The current code rejects pages with too many distinct SKU tokens (catalog filter) and prefers vendor-domain pages. These live-discovered records are flagged with train="live-discovered" so users know they're best-effort pointers, not authoritative. They get overwritten by anything more authoritative (Tier 1 / Tier 2) when those become available.
Many vendor names weren't recognized at all.

"Westermo", "Hirschmann", "Pluribus" etc. returned "Unknown" because the alias tables had only ~25 hardcoded vendors.

Built vendor_registry.py to load vendors.json (134 vendors) once at startup and merge those aliases / domains / portal URLs into every recognition layer. Vendor inference now works across the long tail.
Firmware advisor only worked when a switch row existed in the DB.

The advisor first looked up the switch to get vendor + NOS. After the user wiped the switch DB, every firmware query returned "Switch not found".

The advisor now falls back to alias-based vendor inference when the DB has no row. You can query firmware for any vendor in the registry without first having to populate spec data.

5. What's still missing — honest ceiling

Tier 1 (real release notes) is still 3 vendors.

Allied Telesis, Aruba public RN, Zyxel, FS.com — all the realistic candidates for a 4th Tier-1 fetcher — are either bot-blocked (403) or pure JavaScript SPAs with no static data. Without a cooperating vendor or an LLM extractor we can't move this number.

What this means: the structured "what changed" bullet list will only ever exist for vendors who publish a machine-readable changelog feed. That's an architectural reality at $0 / no-LLM, not a code defect.
Tier 3 live-discovered records have no changelog text.

We return the version pointer + date + URL, but we deliberately do not synthesize bullet diffs — without an LLM that would mean inventing content. Users get the link and decide.

Tier 5 (no-source) is 61 of 134 vendors.

Most are small industrial / specialty brands that don't publish firmware information publicly — only behind a customer login, or only as PDFs gated by partner agreements, or only on JavaScript-rendered pages that need a headless browser. These will stay at Tier 5 until either the vendor changes their publishing model or we add a headless-browser dependency.

Per-model firmware applicability is unused.

The schema has an applies_to_models column but no fetcher populates it. Some vendors ship per-model firmware (one minor release per ASIC) that we currently treat as monolithic per-NOS.

No EOL / End-of-Sale / End-of-Support dates.

Some vendors (Cisco, Arista, Juniper) publish these publicly. Adding them would require a separate scraping path; not yet in scope.

NVD CVE coverage depends on which CPE names have been queried.

18 vendors today, up from 12. Expansion is purely mechanical — add the vendor's CPE pattern to VENDOR_CPES and re-run fetch_firmware.py nvd. Could push to 30+ with diligent work.

6. How to use it

Three entry points, all $0:

  • Web UI: python3 -m streamlit run app.py → Firmware advisor tab → enter a switch model and (optionally) your current version.
  • One model on the CLI: python3 cli.py --firmware "Cisco Catalyst 9300-48P" 17.6.4
  • Fleet (CSV in, CSV out): python3 cli.py --bulk-firmware fleet.csv --out results.csv — runs every row through the advisor and writes 14 columns per row including severity-broken-down CVE counts and the recommended-minimum patched version.

To improve coverage:

  • Add a Tier-1 fetcher for a vendor with a clean public source — write a class in firmware_fetchers.py following the MikroTik / Cumulus / Ubiquiti pattern. ~50–150 lines.
  • Expand NVD coverage — add the vendor's CPE pattern to scrapers/nvd_fetcher.py VENDOR_CPES and run python3 fetch_firmware.py nvd.
  • Pre-warm live discovery — python3 prefetch_firmware.py walks the registry and tries to cache a Tier 3 record for every vendor that has one publicly available.
