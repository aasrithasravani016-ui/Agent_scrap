1. How a spec lookup works (the pipeline)

You enter a switch model. The agent walks five sources in order, cheapest first. The first source that returns a usable record wins. The remaining steps are only triggered if needed.

1. Vendor inference (alias map of 134 vendors)

2. Local cache (switches.db)
→
3. Auto-enrich if sparse
→
4. Live web fallback
→
5. Cache for next time
StepWhat it doesTypical time
1. Vendor inference Aliases / domains from vendors.json (134 entries) map any query to the right canonical vendor — even when the model is typed bare ("CRS326-24G-2S+RM" → MikroTik; "TEG-30284" → TRENDnet). < 1 ms
2. Local cache SQLite lookup — exact SKU → exact model → substring → fuzzy. If a record is found, return it immediately. Includes pre-scraped models and anything previously fetched live. ~1 ms
3. Auto-enrich If the cached row is sparse (fewer than 2 useful fields), quietly trigger a live fetch in the background, merge new fields into the row (never overwriting existing values), persist back, return the enriched record. ~4 s (one-time per model)
4. Live web fallback Multi-engine search (Startpage → Mojeek → DuckDuckGo) → fetch top 3 vendor-domain pages in parallel (HTML + PDF) → parse spec tables → merge by source confidence → cache. ~4 s first hit, then ~1 ms
5. Cache write Every successful live lookup writes a row into switches.db so the next query for the same model is instant. < 1 ms

2. The five data sources

SourceWhat it isUsed for
Local SQLite cache
data/switches.db		 Single-file SQLite database. Schema: 25+ structured columns (vendor, model, ports, capacity, PoE, layer, NOS, …) plus a JSON column for extra fields from the datasheet. Every query starts here. Cached responses return in ~1 ms.
Hand-curated seed
seed_data.json		 Optional starter file — a list of well-known switches typed by hand to bootstrap a fresh install. The agent works fine with an empty seed too. Bootstrap only. Auto-loaded into switches.db on first run via build_db.py.
Per-vendor scrapers
scrapers/ package, 18 modules		 Vendor-specific Python modules that visit a vendor's catalog or sitemap and pull every switch product into the local DB. Each scraper subclasses BaseScraper and implements discover_models() + extract_model(). Bulk population. Run python3 run_scrapers.py once and the cache fills up for the working vendors.
Live web fallback
live_extract.py		 Free multi-engine search + per-source extraction + confidence-weighted merge. The "any vendor, any model" path — works for the 116 vendors without a hand-built scraper, and as a backstop for the scrapers that fail. Cache-miss path. Fetches live, parses HTML/PDF, merges with source-confidence weighting, caches.
Vendor registry
vendors.json + vendor_registry.py		 Canonical list of 134 switch vendors with aliases, website, country, category, notes. Loaded once at startup; merged into every vendor-recognition layer (agent aliases, search-engine vendor index, extraction domain index, login-portal map). Without this every long-tail vendor would say "Unknown" or be misattributed to a competitor URL.

4. The live web fallback — architecture

This is the engine that makes the "any vendor, any model" promise true. Six stages, all within a 9-second budget per query.

StageWhat happens
Search Query is sent to Startpage → falls through to Mojeek if no results → falls through to DuckDuckGo. (DDG hard-blocks scraping with HTTP 202 — that's why it's last.) Free, no API keys.
Rank Each result is scored by source confidence: vendor-domain PDF = 1.00, vendor HTML = 0.85, random PDF = 0.70, reseller = 0.50. Top 3 by confidence proceed.
Fetch in parallel The 3 chosen sources are fetched concurrently with a 5-second timeout each. Network is the bottleneck — parallelism keeps us inside the deadline.
Filter Pages are rejected if they look like multi-product catalogs (more than ~12 distinct SKU-like tokens) — catalog data can't be safely attributed to a single model.
Parse HTML → BeautifulSoup spec-table extraction; PDF → pdfplumber for tables + pymupdf for text. Comparison PDFs with multiple model columns are read from the column whose header matches the queried model.
Merge + cache Multi-source merge with voting — where sources disagree, the higher-confidence one wins, and matching values across sources boost confidence. Mapped fields go in structured columns; everything else goes in the extra_specs JSON column. Result written to the cache before returning.

The same pipeline also extracts the product image (og:image → JSON-LD image array → ranked <img> tags, with logo/sprite filtering).

5. How specs are shown in the UI

Datasheets vary wildly — some have 5 fields, some have 80. The UI adapts:

  • One-line summary built from real data: "48 × 1G access switch · L3 · PoE+ 740 W · 208 Gbps switching". Each piece is optional — only added when the underlying field exists. Never invented.
  • 3-4 headline metric cards chosen dynamically from populated fields (Ports / Capacity / PoE / Layer / NOS / Role). No empty cards — if a field is missing, the metric tile is simply not shown.
  • 7 sectioned panels route the datasheet's extras into the right place: Ports & connectivity · Performance · PoE & power · Layer & features · Software & lifecycle · Physical & environment · Standards & compliance. Empty sections are hidden.
  • "Other datasheet details" is a collapsed catch-all for anything that didn't match a section's keywords. Opt-in expand — not the default view.
  • Product image shown side-by-side with the spec header when one was captured.
  • Live-fetched tag — a small ● live · cached chip appears next to the vendor only when the record was just fetched live (not on local-cache hits).

6. Issues we hit and what we did

DuckDuckGo blocked scraping (HTTP 202).

DDG returns an anti-bot shell instead of results. The live fallback returned nothing.

Switched the search backend to Startpage → Mojeek → DuckDuckGo in fallback order. Both Startpage and Mojeek return real results without API keys.
Multi-product comparison PDFs filled in the wrong column.

A D-Link DES-1008C query returned 1.0 Gbps capacity — but that's the DES-1005C column in the side-by-side comparison table. The parser always read column 1.

PDF parser now matches the queried model name against header rows in comparison tables and reads that column. SKU header rows are skipped, not ingested as specs.
Schema-only extraction dropped most datasheet content.

Anything that didn't fit our structured columns (Standards, Dimensions, MAC Address Table Size, RAM Buffer, Power, LED Indicators, etc.) was silently discarded — even though the parser already extracted it.

Added an extra_specs JSON column. The UI routes each leftover field into the section it belongs to (Performance / Physical / Standards / PoE…) so the datasheet is shown inline, not just linked. For a D-Link DES-1008C this turns "Status: active" into a full sectioned spec card with 73 extra fields.
Multi-product catalog PDFs contaminated single-model results.

A TRENDnet TEG-30284 query picked up "12 ports / 240 Gbps" from a 50-model TRENDnet catalog PDF that happened to mention TEG-30284 alongside dozens of other SKUs.

Page-level filter: a fetched page is rejected if it contains more than ~12-25 distinct SKU-like tokens. URLs that name the queried model are fast-pathed through. No more cross-contamination from catalogs.
Wrong vendor inferred from competitor-mention URL.

A ConnectX-6 query (an NVIDIA NIC, not a switch) returned a Cisco-attributed empty record because the search ranked a Cisco doc that mentioned ConnectX higher than NVIDIA's own page.

Vendor source selection is now confidence-weighted (vendor PDFs > vendor HTML > reseller pages). Added explicit NIC/DPU rejection so ConnectX/BlueField/Intel X5xx/X7xx don't get shoehorned into the switch schema. Plus 134-vendor registry so long-tail names aren't "Unknown".
Sparse cached rows showed a wall of em-dashes.

A Dell S5248F-ON query returned a card with nothing populated except "Status: active" — the seed row had just vendor + model + status.

Auto-enrich on read: if a cache hit has fewer than 2 useful fields, the agent quietly runs a live fetch, merges new fields in (never overwriting), writes the enriched row back, and returns the full card.
PDF libraries weren't installed → 0 fields from every PDF.

Fetchers were correctly downloading datasheets but parsing always returned empty — pdfplumber and pymupdf weren't on the machine.

Listed in requirements.txt; first install picks them up.
Streamlit cross-thread SQLite error.

Streamlit caches the agent via @st.cache_resource and reruns the script on different threads when widgets change. SQLite refused the cross-thread connection.

Opened the connection with check_same_thread=False. The agent's main path is read-only so cross-thread sharing is safe.
Image extractor grabbed brand logos and sprites.

The og:image tag often points at a logo, a sprite, or a placeholder rather than the actual product photo.

Added a logo/sprite/placeholder filter that rejects URLs containing patterns like logo, sprite, opengraph, placeholder, etc. before accepting an image.
TP-Link's product catalog was JavaScript-rendered.

The original scraper hit a JS page and found zero product links — the entire TP-Link catalog was invisible.

Rewrote the scraper to use TP-Link's sitemap (tp-link.com/us/sitemap.xml) instead. ~125 product URLs reliably enumerated; no JavaScript needed. Same pattern would work for other JS-rendered vendor catalogs.
UI alternated between em-dashes and a 73-row dump.

The metric grid showed empty tiles for sparse models, and the "additional details" expander dumped every extra field in one giant unsectioned table.

Replaced fixed 8-metric grid with an adaptive 3-4 cards (only populated metrics shown). Datasheet extras are now routed into 7 named sections. Empty sections are hidden. Anything that didn't match a section's keywords goes in a single collapsed catch-all.
Big marketing hero distracted from the search box.

The Streamlit UI was opening with a full-width blue gradient banner; the search input was below the fold.

Stripped to minimal: small wordmark + centered hero text ("Find any switch in seconds.") + the search input. White background, no banner.

7. What's still missing — honest ceiling

15 of 18 scrapers are thin or blocked.

Only 6 pull real bulk catalog data. The other 12 (9 thin + 3 blocked) rely on the live fallback for any actual content. This isn't a bug — it's vendor publishing reality. Fixing the thin ones means rewriting per-vendor selectors (the TP-Link sitemap approach is the template); fixing the blocked ones means adopting a headless browser or paid bypass service.

Live fallback can't parse pure-JavaScript product pages.

If a vendor's product page renders entirely client-side (Zyxel, modern Ubiquiti techspecs, some D-Link), the parser sees an empty HTML shell. The PDF datasheet is the workaround — usually the live search finds it on a vendor or reseller domain.

No EOL / End-of-Sale / End-of-Support dates.

Cisco, Arista, and Juniper publish these publicly. A per-vendor EOL scraper would be a separate path; not yet in scope.

No transceiver / optic compatibility matrices.

"What 100G optic works in this port?" — each vendor publishes a separate PDF. Per-vendor scraping work.

No pricing / availability.

Reseller-only data, often dynamic, often partner-login-gated.

Image quality varies.

The extractor picks the first og:image that passes the logo/sprite filter. Sometimes that's a side angle, sometimes the hero shot. A per-vendor image-rule system would tighten this but isn't built.

Five registry entries have website: null.

Chilink IoT, LINKOH, Scomp Enterprise, Wanglink, Xexagon — recognised by name but won't get domain-priority weighting in search ranking until their canonical URL is confirmed.
