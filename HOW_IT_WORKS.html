Three independent data layers feed answers:

  • Spec layer (vendor scrapers + live fallback) — ~18 per-vendor scrapers for catalog pages, plus a free-web search fallback (Startpage → Mojeek → DuckDuckGo) that fetches and extracts specs for models we've never seen, then caches them.
  • Security layer (NIST NVD) — works for every major enterprise vendor, including the ones whose own portals are login-gated. Updated daily by NIST.
  • Changelog layer (vendor scrapers) — full "what's new / what's removed" notes for vendors that publish them openly (MikroTik, Ubiquiti, NVIDIA Cumulus).

1b. What is a "scraper", in plain English?

A scraper is a small Python program that visits a public web page (like Cisco's switch catalog), reads it the way a human would, and pulls out the structured pieces — model name, port count, switching capacity, PoE budget, datasheet link — into a clean row in our database.

Think of it as "copy-paste, but automated and consistent". A human could open cisco.com/.../catalyst-9300, eyeball the table, and type the numbers into a spreadsheet. A scraper does the same thing but in milliseconds, for every model, the same way every time.

Each vendor needs its own scraper because every vendor's website is laid out differently. Cisco uses one HTML pattern, Juniper uses another, MikroTik publishes plain-text changelogs, Ubiquiti has a JSON API. One generic crawler can't read all of them.

The three things every scraper does

1
Discover. Visit the vendor's catalog index page and list every product URL on it. Example: open arista.com/en/products/eos and find every link that leads to a switch model page.
2
Extract. For each model URL, fetch the page, locate the spec table or datasheet PDF, and pull out the fields we care about (ports, speed, PoE, switching capacity, layer, NOS, etc.). HTML parsing uses BeautifulSoup; PDF parsing uses pdfminer.
3
Save. Write each model as one row in the local SQLite database (switches table), keyed by (vendor, model). If the row already exists, update only the fields the scrape found — never overwrite good data with empty fields.

What makes them reliable

  • Polite by design. Each scraper waits at least 1 second between requests to the same vendor's site so we never look like a denial-of-service attack.
  • Cached on disk. Every page fetched is saved under data_cache/. Re-runs hit the cache, not the network — fast, free, and friendly to the vendors.
  • Honest output. If a field isn't on the page, the scraper writes NULL. It does not guess. (Hence the agent's tagline: "no LLM, no fabrication.")
  • Per-vendor schema awareness. Each scraper knows that, say, Arista calls it "Switching Capacity" while MikroTik calls it "Switching Fabric" — both map into our one canonical switching_capacity_gbps column.

"Scraper" vs "fetcher" vs "live extractor" — they're related

3. What doesn't work yet, and why

Limited Full release notes for Cisco / Juniper / Arista / HPE Aruba / Dell / Fortinet

What's missing: the bullet-level "added X, fixed Y, removed Z" changelogs.

Why: these vendors put their PDF release notes behind a customer login portal (HPE Passport, Cisco CCO, Juniper Support Portal, Arista Support, Fortinet FortiCare). Their public pages are also Akamai-protected and return HTTP 403 to anything that isn't a real logged-in browser session.

What you still get today: the full CVE / security view via NIST NVD (which is what most upgrades are actually driven by). So the headline question — "is this version vulnerable and what version fixes it?" — is answered.

Not covered Legacy & long-tail switch vendors

These have scraper code stubs but no CVE / changelog data: HP ProCurve / Comware, Huawei VRP, Extreme Networks EXOS, Brocade / Ruckus FastIron, H3C, Lenovo, Edgecore, Zyxel, Ruijie, TRENDnet, TP-Link Omada, Netgear smart switches.

Why: two reasons depending on the vendor —

  • Not standardized in NVD's CPE dictionary. A vendor can't be queried by CPE if NIST has no canonical vendor:product string for them (HP ProCurve, Huawei VRP, Extreme EXOS are all examples).
  • Consumer-grade lifecycle. Netgear / TP-Link / TRENDnet publish firmware per product page, not as a structured changelog. Scraping each model page is per-SKU work.

Out of scope Anything requiring a vendor account

Downloading actual firmware binaries, opening TAC cases, pulling per-account licensing or warranty data — all of these require the user's own credentials and the agent intentionally doesn't try.

5. The pipelines — what happens when you ask a question

There are two pipelines, one per tool in the UI.

Pipeline A — Search specifications

Walkthrough of asking Cisco Catalyst 9300-48P:

1
You type the model (or a comparison like compare C9300-48P vs EX4400-48P). The query goes to SpecAgent.lookup().
2
The agent pulls candidate tokens out of your text — vendor name ("cisco"), model-shaped strings ("9300-48P", "C9300-48P", "Catalyst"). Vendor aliases like "jnpr" → Juniper or "ubnt" → Ubiquiti are normalized here.
3
Local SQLite first. A 4-stage lookup (exact SKU → exact model → substring → fuzzy) against the switches table. If found → answer in a few milliseconds.
4
If not found → live web fallback. live_search.py queries Startpage, Mojeek, then DuckDuckGo (free, no API key) for "vendor model datasheet". Vendor-domain results are prioritized.
5
It opens the top hits. live_extract.py fetches each candidate URL (HTML or PDF), parses with BeautifulSoup or pdfminer, and runs vendor-aware field extractors (the same logic used in the per-vendor scrapers).
6
It merges sources. When 2-3 pages all say "48 ports" but only one mentions "PoE+ 740 W", we take the union with a confidence score. Conflicting numeric fields are resolved by picking the vendor-origin source first, datasheet PDFs second.
7
The result is cached. Written back to switches. Next time the same model is asked, it answers in milliseconds from cache — no web fetch.
8
The UI renders a structured product card — headline metrics (ports / capacity / PoE / layer), feature chips, grouped spec tables (Performance, PoE & power, Software, Physical, Standards), and the datasheet link.

Pipeline B — Firmware advisor

Walkthrough of the original screenshot's query: Aruba + firmware 10.08.1000.

1
You type a vendor name and a version in the Streamlit UI. The form sends both strings to SpecAgent.firmware_advise().
2
The agent recognizes the vendor. It runs the text through an alias map ("aruba" / "hpe" / "hp" → HPE Aruba; "jnpr" → Juniper; etc.). If the text also contained a model token (like 9300-48P) it would try a model lookup first to pin the exact Network OS.
3
It picks the right NOS for the vendor. Each vendor has a default NOS (HPE Aruba → ArubaOS-CX, Cisco → IOS-XE, Juniper → Junos). This is what determines which CVE bucket to query.
4
Layer A: query the security database. advisories_for_version("HPE Aruba", "ArubaOS-CX", "10.08.1000") pulls every cached CVE for that NOS and runs each one's affected version range against the user's version. NOS labels are normalized (so the switch table's "ArubaOS-CX" and NVD's "AOS-CX" both match).
5
It does real version math. A vendor-agnostic version parser tokenizes 10.08.1000 into comparable parts and checks whether it falls inside ranges like [10.08.0000, 10.08.1070). Same parser handles Cisco's 15.2(7)E10, Junos's 22.4R3-S2, Arista's 4.31.2F, etc.
6
Layer B: try the changelog database in parallel. If the vendor has open changelogs cached, it also computes a diff between the user's version and the latest known release (new features, security fixes mentioned in notes, bug fixes, deprecations).
7
It fuses both layers into one FirmwareAdvice object and computes recommended_min_version — the smallest firmware version that lifts the user above every matching CVE.
8
The UI renders the result with a severity-colored card per CVE, the affected version ranges, the fix version, and a direct link to the vendor's own advisory page for the full text.

6. Where the intelligence actually lives

There's no LLM in this agent. The "intelligence" is six small, boring, deterministic things that together produce answers no single data source gives you.

Brain 1 Vendor & model understanding

Free-text inputs like "aruba", "jnpr", "ubnt", "mellanox", "TPE-082" map cleanly to canonical vendor names. Model tokens are pulled out of natural sentences. Fuzzy match (difflib) ranks partial spellings.

Lives in agent.py — VENDOR_ALIASES map and _extract_vendor() / _extract_model_candidates().

Brain 2 Cross-vendor version algebra

Every vendor invents their own version format. The agent parses them all into one comparable shape, so "is 10.08.1000 inside [10.08.0000, 10.08.1070)?" gives the right answer whether you're on MikroTik, Cisco IOS, Junos, EOS, or AOS-CX.

Lives in firmware.parse_version() and Version.__lt__().

Brain 3 CVE range matching

Given a CVE's "affected: from X to Y" rules — with all four combinations of inclusive / exclusive bounds — decides whether the user's running version is actually vulnerable. Computes the earliest patched version across all matching CVEs.

Lives in firmware._version_in_range() and minimum_fixed_version().

Brain 4 NOS label normalization

The switches table calls it ArubaOS-CX; NVD calls it aos-cx; Dell rows say OS10 / SONiC. A small variant generator means the user never has to know.

Lives in firmware._nos_match_variants().

Brain 5 Two-source fusion

Most answers stitch together two unrelated datasets (NVD CVE data + vendor changelog data) into one coherent recommendation. When the two disagree about a fix version, the security layer wins (because NVD is the system of record for "is this still vulnerable").

Lives in firmware.advise().

Brain 6 Live-fetch fallback for unknown switches

If a model isn't in the local database, the agent searches the free web (Startpage → Mojeek → DuckDuckGo), opens the top vendor pages and PDFs, extracts specs, scores the merge, and caches the result.

Lives in live_search.py and live_extract.py.

Brain 7 Spec extraction across 18 different vendor websites

Every vendor lays out their spec pages differently — Cisco uses one HTML structure, Juniper uses another, MikroTik publishes plain text, Ubiquiti has a JSON API. Each per-vendor scraper under scrapers/ knows that vendor's quirks and maps its raw page into one canonical schema (port count, switching capacity Gbps, PoE budget W, layer, NOS, etc.). When the page calls a field something we don't have a column for, it's preserved verbatim in extra_specs so nothing is lost.

Lives in scrapers/<vendor>.py (one file each for the 18 covered vendors) and scrapers/base.py (the shared HTTP cache, retry, and upsert plumbing).

7. If you have to sell it — how to explain

The one-sentence pitch

"Tell me what firmware my switch is running, and I'll tell you which security vulnerabilities you're exposed to and exactly which version to upgrade to — for every major enterprise vendor, in one place, free."

Who buys it

  • Network engineers at companies with mixed-vendor fleets (Cisco core, Aruba access, Juniper edge) who currently have to log into 4-5 different vendor portals to do the same check.
  • Security teams doing quarterly vulnerability audits on network gear — today this means an analyst manually cross-referencing CVE bulletins against an asset inventory spreadsheet.
  • MSPs / managed network providers who need a defensible answer for every customer's switch fleet without carrying every vendor's support contract.
  • Procurement / IT planning for the spec-comparison side: "show me 48-port 10G PoE+ switches from any three vendors, side by side, with patch posture."

Why it's different from what they already have

What they're using today Why it's painful What this agent gives them
NIST NVD website You have to know the exact CPE string. Version-range checks are manual. No cross-vendor view. Vendor-name + version in, ranked CVE list out.
Each vendor's own portal (HPE ASP, Cisco CCO, Juniper Support) One login per vendor. Different UI per vendor. No view of the whole fleet. One UI for all of them. Works without a vendor login.
Tenable / Qualys / Rapid7 $10k+/year. Requires agent or scan access to the device. $0. Just needs "vendor + version" — no network reach.
Spreadsheets + email threads Stale within a week. Re-fetch any time with one command.

What makes it defensible

  • The data is real, not hallucinated. Every answer traces to a NIST NVD CVE record or a vendor's own published changelog. There is no LLM in the loop, so there are no fabricated CVE IDs or made-up fix versions.
  • It runs anywhere. SQLite + Python + Streamlit. Laptop, on-prem server, air-gapped network (if you mirror the NVD feed). No cloud lock-in.
  • The cost is zero. No API bills, no per-seat license, no infrastructure. The data sources are public.
  • The intelligence is in the fusion. Anyone can download an NVD dump. The value is the version algebra + NOS normalization + fusion across two data layers + UI that non-CVE-experts can read.

The honest disclaimers

  • "Planning reference only" — this is decision support, not a replacement for a vendor support contract on the production path.
  • Vendors patch out-of-band sometimes; always sanity-check the recommended fix against the vendor's own release notes before a production upgrade.
  • Coverage breadth depends on what NIST NVD has indexed; obscure or very recent CVEs may take 1-3 days to land.
