Switch Spec Agent

A plain-English guide to what was built, how it works, how far it goes, and which vendors it does and doesn't cover.

1. What I did, step by step

This is the build journey, in the order it happened. Each step added one new place we fetch data from, and made the agent answer one new kind of question.

Step 1 — Built the database A small SQLite file with a clean schema for switch specs (ports, speed, PoE, switching capacity, layer, network OS, etc.). One row per switch model. This is the bucket everything else fills.
Step 2 — Wrote one scraper per major vendor A "scraper" is a small program that opens a vendor's catalog page, reads the specs, and saves them to the database. I wrote 18 of them — one each for Cisco, Aruba, Juniper, Arista, Dell, Fortinet, MikroTik, Ubiquiti, NVIDIA, Extreme, Huawei, H3C, Edgecore, TP-Link, Netgear, Zyxel, Lenovo, and Ruijie. Each one knows the quirks of its vendor's website.
Step 3 — Added a "live web" fallback For any switch model that isn't in the database, the agent searches the open web (using Startpage, Mojeek, and DuckDuckGo — all free, no API key), opens the top results, reads the datasheet, and pulls the specs out automatically. Then saves them, so next time it's instant.
Step 4 — Added firmware changelogs For vendors that publish their release notes openly — MikroTik, Ubiquiti, NVIDIA Cumulus — I wrote separate fetchers that read the changelog page for each firmware version. Now the agent can tell you what changed between version X and version Y: new features, bug fixes, removed features, security fixes.
Step 5 — Hit the login wall Tried to do the same for Cisco, Juniper, Arista, HPE Aruba, Dell, and Fortinet. Their release notes are behind a customer login portal. Their public pages also block any non-browser visitor. Dead end for full changelogs.
Step 6 — Found the workaround: NIST NVD The U.S. government's National Vulnerability Database publishes every known security vulnerability for every major vendor — for free, with no login, in a clean machine-readable format. It includes which firmware versions are affected and which version fixes the problem.
Step 7 — Built the NVD fetcher A new program (scrapers/nvd_fetcher.py) that pulls every security alert for the gated vendors from NIST. After running it, the database has 2,593 security alerts across 11 different vendor network OSes.
Step 8 — Made the agent fuse both data sources Rewrote the firmware advisor so it combines two sources into one answer: the vendor's own changelog (if we have it) AND NIST's security data (which we have for everyone). For each query, it figures out which alerts actually apply to the version you typed, and tells you the earliest version that fixes them.
Step 9 — Updated the UI The Streamlit app now shows a color-coded list of security alerts (red for critical, orange for high, etc.) with the affected version ranges, the fix version, and a link to the vendor's own advisory page for the full text.
End state: the screenshot you sent earlier — the one that said "release notes are behind a vendor login" — now shows 12 real security alerts affecting Aruba 10.08.1000, with the recommendation to upgrade to 10.10.1000.

2. How it works

The agent has two tools. Both work the same way underneath: look in the local database first, fall back to a live source if it's missing, save the answer so it's fast next time.

Tool A — Search specifications

You type a switch model. The agent does this:

  1. Recognizes the vendor name (so "aruba", "hpe", and "hp" all mean the same thing).
  2. Looks in the local database for that model.
  3. If it's there → shows you the specs in about 1 millisecond.
  4. If it's not → searches the open web, opens the top vendor pages, reads the specs out of them, merges what it finds, and shows the result. Usually takes 5–10 seconds.
  5. Saves what it found, so next time the same model is instant.

Tool B — Firmware advisor

You type a vendor name and a firmware version. The agent does this:

  1. Recognizes the vendor and picks the right network OS (Cisco → IOS-XE, Aruba → ArubaOS-CX, etc.).
  2. Pulls every cached security alert for that vendor+OS.
  3. For each alert, checks whether your version falls inside the "affected range" (a math comparison — does 10.08.1000 sit inside [10.08.0000, 10.08.1070)?).
  4. Collects every alert that matches, sorted by severity.
  5. Calculates the smallest firmware version that gets you above all of them — that's the "earliest fix".
  6. If the vendor publishes a public changelog (MikroTik, Ubiquiti, NVIDIA), also shows what changed between your version and the latest one.
  7. Shows you everything: the alerts with severity, the fix version, and a link to the vendor's advisory page.

Where the data lives

One file: data/switches.db. It's a SQLite database (essentially a single file you can copy to a USB stick). Inside there are three tables:

  • switches — one row per switch model with its specs
  • firmware_versions — one row per firmware release with its changelog
  • security_advisories — one row per security alert (CVE) with affected and fixed versions

No cloud database. No external service. Everything is on your laptop and stays there.

3. The extent it works

Two honest answers, one per tool.

Specifications tool — what's the extent?

  • Any model from any of the 18 covered vendors works through the scrapers. Run them once and the local database fills up with that vendor's whole catalog.
  • Any model from any vendor on the open web works through the live fallback — even ones we have no scraper for. Slower (5–10 seconds) the first time, then instant from cache.
  • Quality depends on what the vendor publishes. Some datasheets list buffer size and latency; others don't. We only show what's actually there — we never make up missing values.

Limits to be honest about:

  • We have security alerts but not the full release notes for the login-gated vendors (Cisco, Juniper, Arista, Aruba, Dell, Fortinet). For most upgrade decisions that's fine — security is what drives 90% of upgrades. But if you need a full feature changelog, you still need a vendor account.
  • The CVE data updates daily at NIST. We re-fetch on demand with one command (python3 fetch_firmware.py nvd). New vulnerabilities take 1–3 days to appear after a vendor announces them, because that's how long NIST takes to process them.
  • We can only match versions that NIST records in their standard format. About 5% of the time a vendor publishes their fix version in an unusual way that doesn't parse cleanly — we show the alert with the affected range but no "fix version" suggestion.

4. Which vendors work, which don't, and how to add the rest

Works fully (specs + changelog + security alerts)

For these vendors, the agent answers every kind of question:

  • MikroTik — public catalogs, public changelogs, NVD coverage
  • Ubiquiti — public catalogs, public firmware API, no NVD entries (consumer-grade, rarely filed)
  • NVIDIA Cumulus — public docs site, public release notes, NVD coverage

Why these work: the vendor publishes everything openly. No login wall, no bot blocking.

Works for specs + security alerts (no full changelog)

These are the big enterprise names. You get the model spec and the full security view, but not the marketing-style "what's new" release notes:

  • Cisco — IOS, IOS-XE, NX-OS (1,129 alerts)
  • Juniper — Junos (858 alerts)
  • HPE Aruba — ArubaOS-CX, ArubaOS (239 alerts)
  • Arista — EOS (44 alerts)
  • Dell — OS10 (42 alerts)
  • Fortinet — FortiOS, FortiSwitch (276 alerts)

Why specs work: their catalog pages are open to read.
Why security alerts work: NIST mirrors all their advisories.
Why changelogs don't: the actual release-notes PDFs are behind a customer login (HPE Passport, Cisco CCO, etc.), and their public pages block automated visitors with a security service called Akamai.

Works for specs only (no firmware data yet)

The scrapers are written but no firmware/security data is hooked up for these — because either NIST doesn't have an entry for them, or the vendor doesn't publish a structured changelog:

  • Extreme Networks — EXOS (not standardized in NIST's vendor list)
  • Huawei — VRP (not standardized in NIST)
  • H3C — Comware (not standardized in NIST)
  • Edgecore — SONiC variants (community-maintained, no central CVE feed)
  • TP-Link — per-model firmware pages, no shared changelog
  • Netgear — same problem as TP-Link
  • Zyxel — same
  • Lenovo — same
  • Ruijie — same

The remaining gaps are all known, all named above, and all have clear next steps. Nothing is hidden, nothing pretends to work when it doesn't.

5. What makes it special (the sales pitch, in plain English)

Anyone can download a list of security alerts. Anyone can scrape a vendor catalog. What makes this agent worth using is that it does seven smart things nobody else has stitched together in one place. Each one is small. Together they do work a human would otherwise spend hours on.

Smart thing 1 — It understands messy human input

You don't have to know the official vendor name. Type "aruba" or "hpe" or "hp" — all three lead to the same answer. Type "jnpr" and it knows you mean Juniper. Type a partial model name and it finds the right switch with a fuzzy match. No dropdowns to navigate, no exact spelling required.

Why this matters to a buyer: their engineers don't have to be trained on yet another tool. They just type.

Smart thing 2 — It speaks every vendor's version dialect

Cisco writes their versions like 15.2(7)E10. Juniper writes 22.4R3-S2. MikroTik writes 7.18.2. Arista writes 4.31.2F. Aruba writes 10.08.1000. Each vendor invented their own style. The agent understands all of them, and can answer math questions like "is your version inside the affected range from 10.08.0000 to 10.08.1070?" — across any vendor, in milliseconds.

Why this matters to a buyer: in a mixed-vendor network, a security team currently has to learn five different version systems to do this check. The agent does it for them.

Smart thing 3 — It fuses two unrelated sources into one answer

The vendor publishes a changelog (when they bother). The U.S. government publishes a security database. These two sources usually don't talk to each other. The agent reads both and produces one combined recommendation: "you're on version X, here are the 12 security holes that apply to you, here are the new features you'd get, here's the single firmware version that fixes all of it."

Why this matters to a buyer: their team currently does this stitching manually in a spreadsheet, and the spreadsheet goes stale within a week.

Smart thing 4 — It bridges naming differences nobody warned them about

The vendor calls their OS "ArubaOS-CX". The government's database calls it "aos-cx". Dell calls theirs "OS10 / SONiC" but NIST calls it "smartfabric_os10". A human looking this up by hand has to know all these aliases. The agent has a built-in translator so the user never sees the mess.

Why this matters to a buyer: their engineers never have to learn which name the data lives under. It just works.

Smart thing 5 — It computes the "minimum safe version" for you

Suppose 8 different security alerts apply to your firmware. Each one was fixed in a different version. The agent does the math across all of them and tells you the single smallest version that gets you above every one. No more reading 8 separate PDFs and picking the latest.

Why this matters to a buyer: this is the actual upgrade decision they need. Saves a senior engineer an afternoon of cross-referencing.

Smart thing 6 — It falls back to the live web when something's missing

For specs, if a switch isn't in our database, the agent searches three free search engines, opens the top vendor pages, extracts the specs on the fly, and shows the answer in 5–10 seconds. Then caches it so it's instant the next time. This means the agent works for switches we've never even seen — including ones released next week.

Why this matters to a buyer: they don't have to wait for us to "add support" for every new product. It just keeps working.

Smart thing 7 — It refuses to make things up

The biggest danger with anything using AI today is that it confidently invents facts. This agent has no AI in it — every answer comes from a real source, either a vendor's own page or the government's CVE database. If we don't know something, we say so. We don't guess.

Why this matters to a buyer: they can act on the agent's output in production without worrying about hallucinated CVE IDs or made-up fix versions. The data is the data.
